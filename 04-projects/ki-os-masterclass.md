### Modul 01 — Das neue Paradigma

**Video 1:** Kursüberblick + Angebotsstruktur — kein neuer Inhalt.

**Video 2:** Die drei Wellen der KI-Nutzung (Chatbot → Assistent → Agent) — bereits aus dem Webinar bekannt.

**Video 3 (in Bearbeitung):** Was bedeutet das für das eigene Business?
- Kernfrage: Welcher Anteil der eigenen Leistung ist **operativ**, welcher ist **strategisch/kreativ**?
- Konsequenz: Operatives an KI delegieren + neue Fähigkeiten durch KI-Dienste hinzugewinnen
- Menschliche Exzellenz konzentrieren auf: **Strategie, Kreativität, Beziehung** — dort wo KI nicht hinkommt
- Entwicklung: **Einzelkämpfer** (macht alles selbst oder bezahlt andere) → **Stratege mit KI-Team** (denkt, delegiert, beurteilt)
- Bild: Dirigent eines Orchesters — steuert, lässt Claude/KI umsetzen
- Das KI-Team "denkt" mit, der Mensch behält die Kontrolle über Richtung + Qualitätsurteil
- **Tipp: Rückfragen bei delegierten Aufgaben** — nach der Umsetzung die KI fragen, warum etwas so kodiert/umgesetzt wurde — und dabei explizit um eine einfache Erklärung bitten ("erkläre es mir wie einem Zehnjährigen"). Ziel: technisches Verständnis aufbauen ohne Fachwissen vorauszusetzen, besser einschätzen können wie KI arbeitet.
**Video 4:** Die wichtigste Fähigkeit — **Tempo skalieren**
- Nicht: Vollgas ohne Verständnis was dahinter steckt
- Nicht: alles dreifach absichern + winzige Schritte aus reiner Vorsicht
- Beides hat Vor- und Nachteile — die Kunst liegt im Gefühl dazwischen
- Bild: Ferrari im digitalen Raum — wann gibt man Gas, wann tritt man auf die Bremse?
- Diese Intuition entwickelt sich durch das Arbeiten mit dem System — nicht durch Theorie
- Grundhaltung: **neugierig bleiben + mit dem System arbeiten**

**Video 5:** DSGVO Quick Start
- Ansatz: **Privacy by Design** — von Anfang an DSGVO-konform konzipieren
- Technische Innovation vorantreiben, rechtliche Fragen an Spezialisten (Datenschutzbeauftragte, Wirtschaftsanwälte) delegieren
- Personenbezogene Daten schützen: Namen, E-Mails, Adressen, Telefonnummern, Gesundheits-/Finanzdaten — nichts was eine Person identifizierbar macht
- **Sensibles nie direkt in den Chat:** keine Passwörter, keine API-Keys, keine Kundendaten → Platzhalter nutzen
- Bei Kundenprojekten: **AVV** (Auftragsverarbeitungsvertrag) prüfen sobald personenbezogene Daten verarbeitet werden
- Bei Unsicherheit: Experten fragen, nicht selbst in die Verantwortung bringen

**3 Regeln:**
1. Eigenes Business → **Vollgas** — sensibles raushalten, Platzhalter nutzen
2. Platzhalter für Passwörter, Keys, Kundendaten
3. Kundenprojekte transparent gestalten (KI-Einsatz kommunizieren)

**Checkliste vor dem Loslegen:**
1. API-Keys + Passwörter über `.env` — nie direkt in den Chat
2. Keine echten Kundendaten im Projektordner solange unklar ob personenbezogene Daten betroffen
3. Consumer- und API-Produkte unterscheiden können
4. Bei Kundenprojekten: AVV-Thema ist bekannt
5. Bei Unsicherheit: wissen wen man fragt


### Modul 02 — Dein Setup

**Video 10: Tokenverbrauch verstehen — `/context`**

- `/context` zeigt den aktuellen Tokenverbrauch aufgeteilt nach Kategorie:
  - **System prompt** — CLAUDE.md und System-Anweisungen
  - **System tools** — verfügbare Tools (MCP, Bash, etc.)
  - **Memory files** — MEMORY.md und geladene Memories
  - **Skills** — geladene Skill-Definitionen
  - **Messages** — bisherige Konversation dieser Session
  - **Free space** — noch verfügbarer Kontext
  - **Autocompact buffer** — Reserve für automatisches Komprimieren
- Wenn der Kontext zu einem Großteil gefüllt ist: **`/compact`** einsetzen — fasst die Konversation zusammen und gibt Token frei
- Praktisch: Kontext im Blick behalten, bevor das Fenster voll ist — dann kann man selbst entscheiden wann `compact` Sinn macht
- **`ccm` — Claude Code Usage Monitor:** Externes Tool von GitHub-User `maciek-roboblog`. Muss separat installiert werden. Zeigt den Tokenverbrauch / Kosten detaillierter als die eingebauten Befehle.
- **Status Line:** Unter der Terminalzeile lässt sich eine Statusleiste einblenden, die den Tokenverbrauch permanent anzeigt — ohne `/context` aufrufen zu müssen. Aktivieren über `/statusline`. Zeigt:
  - Modellname
  - Kontextverbrauch als Progress Bar + Prozent
  - Sessionkosten in €
  - Tokenanzahl (verwendet / gesamt) — z.B. `30000/200000 tokens`
  - Hinweis: Token-Zählung muss alle drei Komponenten summieren: neue Token + Cache-gelesene Token + Cache-erstellte Token — sonst stimmt die Zahl nicht
- **Pro-Subscription:** 200.000 Token pro Session verfügbar

**Video 9: Kosten im Griff — 6 Hebel**

**Modell-Preise (grobe Orientierung):**
- Sonnet 4.6 (Standard): $3 Input / $15 Output pro MTok
- Haiku 4.5 (günstig): $1 Input / $5 Output — ca. 1/5 des Preises
- Opus 4.6/4.7 (stark): $5 Input / $25 Output

**6 Kostenhacks:**

1. **Modell bewusst wählen** — Haiku für einfache Aufgaben, Sonnet/Opus für komplexe. Wechseln mit `/model`. Standard ist Sonnet — an die Aufgabe anpassen.

2. **`/compact` nutzen** — Fasst die bisherige Konversation zusammen und gibt Kontext-Speicher frei. Sinnvoll wenn eine Session länger wird, spart Token.

3. **CLAUDE.md einmalig gut aufsetzen** — Wer man ist, was man tut, alle Regeln. Wird automatisch jede Session eingelesen → kein manuelles Nachfüttern nötig. Spart Token bei jedem Start. Details: Modul 4.

4. **Skills für wiederkehrende Aufgaben** — Kurzer Befehl statt langer Prompt. Einmal definiert, immer wieder nutzbar. Details: Modul 5.

5. **Neue Session starten** — Eine frische Session kostet nichts. Ein volles Kontextfenster kostet Token. Nach erledigter Aufgabe: neue Session. Wenn Kontext noch gebraucht wird: `/compact` nutzen oder weiterarbeiten — abhängig von der benötigten Detailtiefe.

6. **Aufgaben richtig dimensionieren** — Nicht zu groß (Fehler kosten extra Token), nicht zu klein (Rückfragen sind teuer, da Output-Token teurer als Input-Token). Lieber mehr Infos rein, weniger Rückfragen. **Fazit: Eine klare Aufgabe pro Auftrag.**

**Subscription-Empfehlung (Garrit):**
- Einstieg: **Pro Plan**
- Intensive Nutzung: **Max Plan** — günstiger als pay-per-use via API bei hohem Volumen

**Video 8: Unter der Haube**
- Claude = KI-Modell, läuft auf Anthropic-Servern in der Cloud
- Drumherum: viele Features die wir diesem Modell bereitstellen:
  - Vault (globaler Kontext)
  - CLAUDE.md
  - Commands + Skills
  - Agents + MCPs
- Alles läuft über IDE + Terminal auf der lokalen Hardware — von dort wird mit dem Cloud-Modell kommuniziert
- Bild: Claude (Modell) ist der Kern — alle anderen Schichten des KI OS sind der Rahmen drumherum
- **Harness** = Fachbegriff für dieses Gesamtkonstrukt (dt. Gurt/Geschirr) — fasst KI-Modell + alle Komponenten zusammen
- **Datensicherheit:** Alle Dateien bleiben auf der lokalen Hardware — sie werden nicht auf Anthropic-Servern gespeichert. Anthropic stellt nur die Rechenleistung/Prozesse bereit.
- **Wichtig:** Claude Code ist kein lokales Programm — es "denkt" nicht auf dem Rechner
  - Ablauf: Anfrage wird an Anthropic-Server geschickt → Antwort kommt zurück → Ausführung erfolgt lokal
  - Konsequenzen: Kosten (API-Nutzung), Datenschutz (Konversationsinhalt geht rüber), Workflow (Internetverbindung nötig)
- **Ablauf im Detail:**
  1. Anweisung in Terminal eingeben
  2. Claude Code bündelt: Nachricht + CLAUDE.md + gelesene Dateien + bisherige Konversation
  3. Alles wird via **TLS verschlüsselt** an die Anthropic API-Server übertragen
  4. Anthropic KI-Modell verarbeitet die Anfrage
  5. Antwort wird zurück an den Terminal geschickt
  6. Ausführung lokal: Dateien erstellen, Code ausführen, Bash-Befehle ausführen
- **Was wird an Anthropic übertragen?**
  - Prompts (Eingaben im Terminal)
  - Datei-Inhalte die Claude aktiv liest
  - Tool Outputs (z.B. Ergebnis eines Bash-Befehls)
  - Claudes Antworten / Konversationshistorie
  - CLAUDE.md (wird bei jedem Session-Start übertragen)
- **Was bleibt lokal?**
  - Dateien die Claude nicht aktiv liest
  - Alle anderen Bereiche des Rechners: Browser, E-Mails, Passwörter
  - Code-Ausführung läuft lokal
  - Erstellte Dateien sind nur lokal vorhanden

### Modul 03 — Claude Code bedienen

**Video 1: Vom Prompt zum Briefing**
- Unterschied zwischen LLMs (Large Language Models) und grafischen KI-Tools (ChatGPT, Claude.ai etc.)

**Die drei Arbeitsweisen mit KI:**

| Modus | Beschreibung | Ideal für |
|-------|-------------|-----------|
| **Conversational** | Dialog — Sprache oder Text, gemeinsam zum Ergebnis | Kreative Aufgaben ohne klar definiertes Endergebnis |
| **Single Shot** | Ein perfekt strukturierter Prompt → fertiges Ergebnis | Produktive Aufgaben mit klarem Ziel |
| **Agentisch** | Claude liest Dateien, plant Schritte, führt aus, iteriert im Projektkontext | Komplexe Umsetzungsaufgaben — ich briefe, Claude handelt |

**Das Neue am agentischen Modus:**
- Nicht mehr: Ich schreibe einen Prompt und warte auf Text
- Stattdessen: Ich gebe ein **Briefing** — Claude übernimmt die Umsetzung eigenständig
- Claude arbeitet im Kontext eines Projekts: liest Dateien, plant, führt aus, korrigiert sich
- Rollenwechsel: Ich = Auftraggeber/Stratege — Claude = Umsetzer

**Prompt vs. Briefing — der praktische Unterschied:**

| | LLM / Single Shot | Agentisch (Claude Code) |
|---|---|---|
| **Kontext** | Jedes Mal manuell in den Prompt schreiben | Einmal im Vault hinterlegen — Claude holt sich was es braucht |
| **Promptlänge** | Ellenlang: Rolle + Aufgabe + Format + Anweisungen + Hintergrund | 2–3 Zeilen: "Hol dir X aus Datei Y und löse Z" |
| **Qualität** | Abhängig davon wie gut der Prompt ist | Claude hat mehr Kontext → liefert besseres Ergebnis |
| **Wiederholung** | Jedes Mal von vorne | Kontext ist dauerhaft gespeichert |

**Der größte Hebel: persönlicher Kontext**
- Die Modelle sind mittlerweile so intelligent, dass sie auch aus mittelmäßigen Prompts herauslesen, was gemeint ist
- Was sie aber *nicht* kennen: **wer du bist** — dein Stil, deine Qualität, deine Zielgruppe, dein Warum
- Bild: Supergenie an der Seite — enorm leistungsfähig, aber ohne persönlichen Kontext generisch
- Aufgabe: ein intelligentes System aufsetzen, das diesen Kontext dauerhaft bereitstellt
- Skill-Dateien + Vault sind das Vehikel dafür — einmal aufgebaut, immer verfügbar

**Die 3 Regeln für Briefings (statt "Prompts"):**

> Wir nennen es jetzt Briefings — gleiche Technik, andere Intention.

1. **Kontext lebt in Dateien, nicht im Prompt**
   - Alles was Claude regelmäßig braucht → einmal in Dateien hinterlegen
   - Claude greift jederzeit darauf zu — kein manuelles Einfügen mehr

2. **Was + Warum — nicht Wie**
   - Claude sagt man, was man braucht und warum — nicht wie es umzusetzen ist
   - Das Genie entscheidet selbst über Weg und Format
   - Statt: *"Erstelle eine Tabelle mit 3 Spalten"*
   - Besser: *"Erstelle eine Wettbewerbsanalyse, die mir hilft meine Preispositionierung zu überprüfen"*
   - Format, Struktur, Weg → Claude entscheidet

3. **Referenziere statt kopiere**
   - Nicht: Kontext mühsam in den Chat tippen
   - Stattdessen: auf die Vault-Dateien verweisen — Claude holt sich den Inhalt selbst
   - Das Second Brain (Vault / Markdown-Dateien) = die Referenzdatenbank für alle Briefings

**Wann welcher Ansatz — das Pareto-Prinzip:**

| Anteil | Situation | Ansatz |
|--------|-----------|--------|
| **80 %** | Normale Aufgaben mit Projektkontext | Natürliches Briefing + gute Systemstruktur → einfach abgeben |
| **20 %** | Hochspezifische Ausgaben nötig | Mehr Präzision im Briefing |

**Wann mehr Präzision nötig ist:**
- Bestimmtes Format/Struktur wird vorausgesetzt (z.B. spezifische DJ-Struktur, Plattformformat)
  → Format angeben + **repräsentative Beispiele** für den Zielzustand mitliefern
  → Bei Content-Erstellung: Beispiele/Templates im Vault ablegen, dann per Referenz verfügbar
- Automatisierungen
  → Lösungszustand präzise beschreiben: was muss das System am Ende können, welche Plattformen, welche Zugänge
  → Detailliertes Briefing zum Startzustand — aber den Lösungsweg für die erste Version trotzdem Claude überlassen

**Einmalige Aufgaben ohne Projektordner:**
- Kein Projektkontext, kein Vault-Material — aber trotzdem Qualitätsanspruch
- Dann: **klassischer Prompt**
- Empfehlung: **Meta-Prompting**
  1. In ein klassisches LLM gehen (z.B. Claude.ai)
  2. Per Spracheingabe (Wispr Flow) kurz reinreden was man braucht
  3. Sich den fertigen Prompt vorschreiben lassen
  4. Diesen Prompt dann einsetzen

**Der Shift — weg vom Prompt Engineering:**
- Prompt Engineering war eine Phase — wichtig zu kennen, aber nicht mehr der größte Hebel
- Früher: den perfekten Prompt engineeren → heute: das intelligente System bauen
- **Größter ROI heute:** nicht der perfekte Prompt, sondern die perfekte Umgebung
  - Richtige Inhalte → richtigem Kontext → richtig aufbereitet
  - Vault + Skill-Dateien = das Wissen über *Was* und *Wie*
  - Workspace = die Umgebung in der Claude arbeitet
- Ergebnis: Claude kann direkt hochqualitative Ergebnisse liefern — ohne ellenlange Prompts

**Konsequenz für Christians Setup:**
- `ich.md`, `business.md`, `brand-voice.md` etc. im Vault sind genau das: **einmal gespeicherter Kontext**
- Zielgruppe, Ton, Stil, Hintergrund — Claude holt sich das selbst
- Aufgabe im Chat = kurz + präzise, kein Roman mehr nötig

**Video 6: Fehler vermeiden bei Claude Code**

**Fehler 1 — Die Mammut-Session**
- Problem: Mehrere unzusammenhängende Aufgaben (Newsletter, Präsentation, Analyse, Tabelle) in einer langen Session hintereinander
- Konsequenz: Das Kontextfenster füllt sich mit gemischten Informationen — Newsletter-Kontext "verschmutzt" die spätere Analyse
- Lösung: **Neue Aufgabe = neue Session** (`/clear`) — oder mindestens `/compact` zwischen thematisch getrennten Aufgaben
- Faustregel: **Eine fokussierte 15-Minuten-Sitzung schlägt eine diffuse 2-Stunden-Sitzung**

**Fehler 2 — Das blinde Durchwinken**
- Problem: Claude fragt um Erlaubnis — reflexartiges "Allow" ohne zu lesen
- Der Unterschied zu ChatGPT: dort war der schlimmste Fall ein schlechter Text. Bei Claude Code kann eine Datei überschrieben werden, ein ungewollter Git-Commit entstehen oder ein falscher Bash-Befehl auf dem Rechner ausgeführt werden
- Lösung:
  - Berechtigungsprompt lesen und verstehen was dahinter steckt
  - Diffs anschauen (was wird konkret geändert?)
  - Bash-Befehle verstehen bevor man sie freigibt
  - Git als Sicherheitsnetz nutzen — Commits sind rückholbar
- Grundhaltung: **Du bist der Fahrer. Je mächtiger die Aktion, desto aufmerksamer sein.**

**Fehler 3 — Alles in den Prompt statt in Dateien (Überprompting)**
- Problem: Gesamten Kontext jedes Mal neu in den Prompt schreiben — wie bei ChatGPT, wo das nötig war
- Bei Claude Code ist das kontraproduktiv: alles was wiederkehrt (Hintergrundwissen, Entscheidungsgrundlagen, Rolle, Stil) gehört in Dateien
- Lösung: **`CLAUDE.md` einmalig anlegen** mit Rolle, Kontext und Regeln — Claude liest das automatisch bei jeder Session
- Danach reicht ein Mini-Briefing: *"Schreib mir einen Newsletter über Thema X — Hook, 3 Learnings, CTA"*
- Der Kontext kommt aus der Datei, nicht aus dem Prompt

**Fehler 4 — Zu viel auf einmal verlangen**
- Problem: Claude verführt durch seine Mächtigkeit dazu, riesige Aufgaben auf einmal zu stellen — das Ergebnis wird dünn auf mehreren Ebenen
- Lösung: **Konzeptionell in Phasen arbeiten**
  - Ausgangspunkt: Problem klar beschreiben + relevante Dateien als Hintergrundwissen bereitstellen
  - Ziel: Lösungsideal benennen — der Weg dorthin ist erst mal offen
  - Vorgehen: Aufgabe in kleine Teile/Phasen unterteilen
  - Phase 1: Grundstruktur + Basisdaten erarbeiten → prüfen
  - Phase 2: Nächsten Baustein draufsetzen → prüfen
  - Weiter Stück für Stück — nicht alles auf einmal abgeben
- Bild: Von einem bekannten Problem zu einem unbekannten Lösungsweg — Schritt für Schritt durch einzelne Glieder einer Kette

**Was stattdessen zu tun ist — die 4 Grundprinzipien**

1. **Kontext liefern** — Wichtige Infos in Kontext-Dateien ablegen, zu Sessionstart konkret auf relevante Dateien verweisen
2. **Plan Mode nutzen** — Bei komplexen Aufgaben erst Zielzustand und Umsetzungsplan erstellen lassen, dann freigeben. Bei einfachen Fragen direkt loslegen.
3. **Iterieren** — Schnell eine erste Version erstellen, Feedback geben, bis zum Meilenstein arbeiten, dann prüfen und weiter verbessern
4. **Kontext managen** — Merken wann Ergebnisse stagnieren → `/compact` (Ergebnis sichern, Kontext verdichten) oder neue Session mit frischen Augen eröffnen

**Fazit Video 6 — Die drei Kernfragen vor jeder Session**

> Die Kunst ist nicht, Claude Code in allen Feinheiten perfekt zu beherrschen. Es sind drei Fragen, die zu Beginn jeder Session gestellt werden müssen.

Analogie: Was muss bereit sein, wenn ich den Schlüssel im Zündschloss des Ferraris umdrehe — damit der Motor startet und ich von A nach B fahren kann?

1. **Welche Informationen müssen da sein?** — Kontext bereitstellen bevor die Session startet
