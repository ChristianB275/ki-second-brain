KI OS — Workspace-Sync-Architektur

Stand: 17. April 2026 Version: 1.0 KI-Modell: Claude Opus 4.6 Zielgruppe: KI-Anwender mit Claude Code Erfahrung

Lernziele

  • Verstehen, warum ein KI-Betriebssystem (KI OS) und Workspaces ohne Sync-Mechanismus auseinanderdriften
  • Die Architektur der bidirektionalen Synchronisation zwischen KI OS und Workspaces kennen
  • Die vier Sync-Mechanismen (status.yaml, sync-brain.sh, Git-Pipeline, Server-Cron) im Detail verstehen
  • Den vollstaendigen Session-Ablauf von Start bis Shutdown nachvollziehen koennen

Das Problem: Drift zwischen Wissen und Arbeit

Wer mit Claude Code arbeitet, hat typischerweise zwei Ebenen:

KI OS (das Gehirn)

  • Obsidian Vault mit Strategie, Prioritaeten, Projekt-Stubs
  • Weiss, welche Projekte existieren und wie ihr Status ist
  • Fuehrt Daily Logs, Open Loops, Entscheidungen
  • Dient als Startpunkt jeder Session

Workspaces (die Arbeit)

  • Einzelne Projektverzeichnisse mit Code, Configs, Docs
  • Hier passiert die eigentliche Arbeit
  • Status aendert sich bei jeder Session
  • Haben keine Verbindung zurueck zum KI OS
Das Kernproblem: Wenn in einem Workspace gearbeitet wird, aktualisiert sich der KI-OS-Projektstub nicht automatisch. Nach wenigen Sessions ist das KI OS komplett veraltet — es behauptet z.B. ein Projekt sei bei Phase 1, waehrend der Workspace laengst bei Phase 6 steht.

Beispiel: Drift in der Praxis

Ein Anwender hat ein Server-Projekt mit 7 Phasen. Nach drei intensiven Arbeitssessions im Workspace:

QuelleStatusFortschrittNaechster Schritt
KI OS (Projekt-Stub)gelb10%Phase 1 abschliessen
Workspace (tatsaechlich)gruen85%Nur noch 2 Restpunkte

Das KI OS ist um 6 Phasen veraltet. Jede Session, die mit /start beginnt, arbeitet mit falschen Informationen.

Die Loesung: Bidirektionaler Sync

Die Architektur verbindet vier Komponenten zu einem geschlossenen Kreislauf:

Workspace (status.yaml) ← wird beim Arbeiten aktualisiert ↓ sync-brain.sh ← liest alle status.yaml, erkennt Drift ↓ KI OS Stubs (04-projects/) ← werden bei /start automatisch abgeglichen ↓ git push (bei /shutdown) ← KI OS nach GitHub ↓ Server Cron (*/15 min) ← pullt automatisch von GitHub
Prinzip: Die status.yaml im Workspace ist die Single Source of Truth. Das KI OS liest diese Quelle und aktualisiert sich selbst. Nicht umgekehrt.

Mechanismus 1: status.yaml

Jeder Workspace bekommt eine status.yaml im Root-Verzeichnis. Diese Datei enthaelt genau die Felder, die das KI OS fuer seinen Projektstub braucht:

projekt: webshop-relaunch
status: gelb
fortschritt: 45
letztes-update: 2026-04-17
naechster-schritt: "Payment-Integration testen"
meilensteine:
  - name: "Grundgeruest"
    status: gruen
  - name: "Produktkatalog"
    status: gruen
  - name: "Payment-Integration"
    status: gelb
  - name: "Go-Live"
    status: rot

Felder im Detail

FeldTypBeschreibung
projektStringEindeutiger Projektname in kebab-case — muss mit dem KI-OS-Stub uebereinstimmen
statusEnumrot (Blocker), gelb (in Arbeit), gruen (laeuft/fertig)
fortschritt0–100Geschaetzter Gesamtfortschritt in Prozent
letztes-updateDatumWann wurde zuletzt im Workspace gearbeitet
naechster-schrittStringEinzeiler: was als naechstes ansteht
meilensteineListeJeder Meilenstein mit Name und Ampel-Status
Wann wird status.yaml aktualisiert? Am Ende jeder Workspace-Session — entweder manuell oder durch Claude beim Session-Shutdown. Der KI-Assistent schlaegt Aenderungen vor, der Anwender bestaetigt.

Mechanismus 2: sync-brain.sh

Ein Shell-Script im KI OS liest alle status.yaml-Dateien aus den Workspaces und vergleicht sie mit den KI-OS-Projektstubs:

$ bash scripts/sync-brain.sh

=== KI OS Sync Check ===

OK     webshop-relaunch   — gelb/45%
OK     api-backend        — gruen/90%
DRIFT  iot-dashboard      — Workspace: gruen/70% | Brain: gelb/30%
       Workspace sagt: MQTT-Bridge konfigurieren
OK     dokumentation      — gruen/100%

1 Abweichung(en) gefunden — KI OS Stubs aktualisieren.

Was das Script macht

  1. Iteriert ueber alle Workspace-Verzeichnisse und sucht status.yaml
  2. Parsed die YAML-Felder (status, fortschritt, letztes-update) mit Standard-Shell-Tools
  3. Findet den passenden KI-OS-Stub ueber das projekt:-Feld im YAML-Frontmatter
  4. Vergleicht Workspace-Status mit Stub-Status
  5. Gibt OK (synchron) oder DRIFT (abweichend) pro Projekt aus
Kein externer Parser noetig: Das Script verwendet nur grep, awk und head — funktioniert auf jedem Unix-System ohne Python oder yq.

Mechanismus 3: Git-Pipeline (KI OS nach GitHub)

Das KI OS liegt als privates GitHub-Repository. Jede Session-Beendigung (/shutdown) fuehrt automatisch drei Schritte aus:

  1. git add -A — alle Aenderungen im Vault stagen
  2. git commit — mit Zusammenfassung der Session als Nachricht (nur wenn Aenderungen vorhanden)
  3. git push — zum privaten GitHub-Repository

Warum Git und nicht Cloud-Sync?

Git-Vorteile

  • Vollstaendige Versionshistorie aller Aenderungen
  • Diff-basiert: nur Aenderungen werden uebertragen
  • Deploy-Keys ermoeglichen granularen Zugriff (read-only moeglich)
  • Standard-Tooling, kein proprietaerer Sync-Client
  • Funktioniert offline, synchronisiert bei Verbindung

Cloud-Sync-Nachteile

  • Keine Versionshistorie (nur aktueller Stand)
  • Sync-Konflikte bei gleichzeitigem Zugriff
  • Abhaengigkeit von proprietaerem Client (Google Drive, iCloud)
  • Kein granularer Zugriff fuer Server (alles oder nichts)
  • Latenz: Cloud-Sync kann Minuten dauern

Mechanismus 4: Server-Sync (Cron-Pull)

Ein Automatisierungsserver (z.B. mit n8n, Home Assistant oder einem einfachen Linux-Server) zieht das KI OS regelmaessig von GitHub:

Einrichtung

  1. Deploy-Key generieren: SSH-Schluessel am Server erzeugen (ssh-keygen -t ed25519)
  2. Deploy-Key bei GitHub hinterlegen: Oeffentlichen Schluessel als Read-Only Deploy-Key im Repository eintragen
  3. SSH-Config: Host-Alias anlegen, der den Deploy-Key fuer github.com verwendet
  4. Clone: Repository am Server klonen (einmalig)
  5. Cron-Job: Alle 15 Minuten git pull --ff-only ausfuehren

Sync-Script am Server

#!/usr/bin/env bash
# Cron: */15 * * * *
set -euo pipefail

cd "$HOME/server/inbox/ki-os"
OUTPUT=$(git pull --ff-only 2>&1)

# Nur loggen wenn neue Commits ankommen
if ! echo "$OUTPUT" | grep -q "Already up to date"; then
    COMMITS=$(echo "$OUTPUT" | grep -c "^ " || true)
    echo "{\"ts\":\"$(date -u +%FT%TZ)\",\"type\":\"ki-os-sync\",\"commits\":$COMMITS}" >> events.jsonl
fi
Ergebnis: Der Server hat innerhalb von maximal 15 Minuten nach einem /shutdown den aktuellen KI-OS-Stand. Workflows koennen auf Projektdaten zugreifen, Dashboards koennen Ampeln anzeigen, Alerts koennen auf Statusaenderungen reagieren.

Ablauf: Session starten

Beim Start einer KI-OS-Session (z.B. mit dem Befehl /start) laeuft automatisch folgender Ablauf:

/start — Automatischer Ablauf
7 Schritte
  • 01
    Workspace-Sync pruefensync-brain.sh ausfuehren. Bei Abweichungen: KI OS Stubs automatisch aktualisieren, committen, pushen.
  • 02
    Kontext ladenPersoenliches Profil und Geschaeftsstrategie aus dem Vault lesen.
  • 03
    Prioritaeten ladenAktuelle Wochenprioritaeten und offene Aufgaben.
  • 04
    Open Loops pruefenWartende Aufgaben und offene Entscheidungen. Loops aelter als 2 Wochen werden markiert.
  • 05
    Inbox pruefenUngefilterte Notizen und Ideen, die noch eingeordnet werden muessen.
  • 06
    Letzte Daily Logs lesenDie letzten 2 Tages-Protokolle fuer Kontext zur aktuellen Arbeit.
  • 07
    Dashboard anzeigenKompakte Uebersicht: Top 3 Prioritaeten, Open Loops, Inbox-Stand.
Entscheidend ist Schritt 1: Bevor irgendein Kontext angezeigt wird, werden die Workspaces abgeglichen. So arbeitet das KI OS immer mit dem tatsaechlichen Stand — nicht mit dem, was es zuletzt gespeichert hat.

Ablauf: Im Workspace arbeiten

Waehrend der Arbeit in einem Workspace passiert nichts Besonderes in Bezug auf den Sync. Die status.yaml wird erst am Ende aktualisiert. Der KI-Assistent:

  • Arbeitet ausschliesslich im Workspace-Verzeichnis
  • Fuehrt Aufgaben aus, erstellt Dateien, aendert Konfigurationen
  • Committet und pusht Aenderungen im Workspace (normaler Git-Workflow)
  • Aktualisiert status.yaml am Ende der Session mit dem neuen Stand

Beispiel: status.yaml vor und nach einer Session

Vor der Session

  • status: gelb
  • fortschritt: 45
  • naechster-schritt: "Payment testen"
  • Payment-Integration: gelb

Nach der Session

  • status: gruen
  • fortschritt: 70
  • naechster-schritt: "Staging-Deploy"
  • Payment-Integration: gruen

Ablauf: Session beenden

Der /shutdown-Befehl im KI OS fuehrt den vollstaendigen Abschluss durch:

  1. Prioritaeten und Open Loops aktualisieren
  2. Ampel-Vorschlag pro beruehrtem Projekt (der Anwender bestaetigt oder korrigiert)
  3. Daily Log schreiben (was passiert ist, Entscheidungen, naechste Session)
  4. Lernprotokoll ergaenzen (falls Korrekturen in der Session stattfanden)
  5. git add -A, git commit, git push — KI OS geht nach GitHub
Geschlossener Kreislauf: Nach dem Shutdown ist alles gesichert. Der Server holt sich den neuen Stand innerhalb von 15 Minuten. Die naechste /start-Session beginnt mit aktuellem Wissen.

Vorher / Nachher

Ohne Sync-Architektur

  • KI OS driftet nach jeder Workspace-Session weiter ab
  • /start zeigt veraltete Informationen
  • Manuelles Nachpflegen noetig (wird vergessen)
  • Server hat keinen Zugriff auf Projektdaten
  • Vertrauen in die Statusanzeige geht verloren

Mit Sync-Architektur

  • KI OS gleicht sich automatisch ab beim naechsten /start
  • Dashboard zeigt immer den echten Stand
  • Kein manuelles Nachpflegen noetig
  • Server hat binnen 15 Minuten aktuelle Daten
  • Versionierte History aller Zustandsaenderungen

Glossar

KI OS
Das zentrale Wissenssystem (Obsidian Vault) mit Strategie, Projektstubs, Daily Logs und Entscheidungen.
Workspace
Einzelnes Projektverzeichnis mit Code, Konfiguration und Dokumentation. Hier wird die eigentliche Arbeit erledigt.
Projekt-Stub
README.md mit YAML-Frontmatter im KI OS, die den Status eines Projekts zusammenfasst (Ampel, Fortschritt, naechster Schritt).
status.yaml
Datei im Workspace-Root, die den aktuellen Projektstatus maschinenlesbar abbildet. Single Source of Truth.
Drift
Zustand, in dem der KI-OS-Stub und der tatsaechliche Workspace-Status voneinander abweichen.
Deploy-Key
SSH-Schluessel, der einem Server Zugriff auf ein spezifisches GitHub-Repository gibt (read-only oder read-write).
Ampel-System
Farbcodierung fuer Projektstatus: rot (Blocker), gelb (in Arbeit), gruen (laeuft/fertig).
Fast-Forward Pull
git pull --ff-only — akzeptiert nur neue Commits, bricht bei Konflikten ab. Sicherster Pull-Modus.
